シナリオ 1 低速クエリ（Slow Query）

device-api のデバイス一覧クエリに人工的な遅延を追加。Application Signals でレイテンシP99スパイクを観察する。

CloudWatchで確認すること：
  • Application Signals → サービス → device-api → レイテンシ グラフの P99 スパイク
  • Application Signals → サービスマップ → netwatch-ui → device-api の矢印が赤くなる
  • X-Ray トレース → タイムライン表示 → device-api の span が長い
  • Log Insights: filter @message like "slow_query" | stats avg(duration_ms) by bin(1m)
有効後: /devices ページを開くと遅延が発生します
シナリオ 2 エラー注入（Error Injection）

device-api の一定割合のリクエストを強制的に HTTP 500 エラーにする。エラー率の急上昇を検知する。

CloudWatchで確認すること：
  • Application Signals → サービス → device-api → エラー率 グラフの上昇
  • CloudWatch Logs → /aws/containerinsights/obs-poc/application → ERROR ログ検索
  • Log Insights: filter levelname = "ERROR" | stats count() by bin(1m)
  • アラーム設定例: エラー率 > 5% で SNS 通知
有効後: /devices を複数回リロードするとエラーが発生します
シナリオ 3 アラートストーム

alert-api から大量のアラートを短時間で生成。ログボリュームの急増とアラート疲労を体験する。

CloudWatchで確認すること：
  • CloudWatch Logs → ログのバイト数メトリクスが急増
  • Log Insights: filter event = "alert_storm" | stats count() by severity
  • Container Insights → ネームスペース demo-ec2 → alert-api のCPU使用率
  • アラーム設定例: ログ取り込み量 > 閾値 で通知
開始後: /alerts ページでアラートが増加します
RUM検証 JavaScript エラー発生（CloudWatch RUM 検証用）

ブラウザで意図的な JS エラーを発生させ、CloudWatch RUM の JS エラー検知を確認する。RUM を設定済みの場合のみ意味があります。

CloudWatch RUMで確認すること：
  • RUM → JS Errors → エラー件数が増加するか
  • RUM → Sessions → このページへのアクセスが記録されるか
  • エラーメッセージ: NetWatch RUM test error が見えるか
CloudWatch RUM を設定していない場合は無視してください。ブラウザのコンソールにエラーが出るだけです。
